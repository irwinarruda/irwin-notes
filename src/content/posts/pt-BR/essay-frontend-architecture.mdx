Sempre me incomodei com a falta de padrões na criação de aplicações frontend com os frameworks modernos. Quando comecei a desenvolver, já eram 6 anos de vida do React e as pessoas ainda discutiam se deveriam utilizar Redux ou Context API para gerenciar estados. Foi só recentemente que as documentações ficaram melhores, as bibliotecas se estabilizaram e um caminho claro se abriu para ser seguido.

Os grandes frameworks parecem estar convergindo para um padrão "ótimo" no desenvolvimento de aplicações web. O **Vue** introduziu o _ref_ e _hooks_, **Svelte** criou _Runes_ e o **SolidJS** sempre foi muito semelhante ao **React**. Me parece que a evolução e dos principios core do React moveram todo o mercado para lugares parecidos.

Alguns padrões primitivos foram criados e aperfeiçoados ao longo do tempo, todos os frameworks possuem o conceito de _State_, _Computed_, _Actions_ e _Effects_. Esses padrões me fizeram coçar a cabeça e estudar sobre o assunto. Minha proposição e o motivo desse ensáio é de que, talvez usando esses **axiomas**, seja possível derivar um modelo de arquitetura universal, ou seja, uma arquitetura de frontend independente de bibliotecas!

<PostCallout>
  <span>Mas arquitetura no frontend é mesmo importante?</span>
</PostCallout>

Na literatura, o frontend é normalmente tratado como uma camada. Ele é _apenas_ uma parte do sistema, um detalhe de implementação. E de fato, _muitas das vezes_, o frontend é somente uma maneira de conectar um usuário ao que gera valor (à finalidade da aplicação), que fica num backend.

Por mais que possa ferir o coração de muitos _frontenders_, isso é verdade! Existem sistemas CRUD e que não precisam de uma arquitetura robusta no frontend. Testes e manuteniblidade não são necessários. Substituir essa UI por outra não geraria impactos grandes no negócio.

É claro que essa é somente uma parte da história, na verdade, frameworks como React e Angular só foram criados pois a UI precisava evoluir para gerar uma boa experiência aos usuários. A partir dai, o frontend começou a ter importância negocial, começou a gerar valor. Porém, o preço de uma experiência inovadora é complexidade no software. Com isso, o frontend para de ser um detalhe de implementação e vira também um fim em si, e por isso faz sentido se preocupar com sua arquitetura. 

<PostCallout>
  <span>Então como devemos gerenciar a complexidade de uma UI interativa com as exigências do mercado?</span>
</PostCallout>

Existem muitas formas de organizar um software, e elas foram extensivamente documentadas por grande autores, porém como o Uncle Bob deixa explícito em [The Clean Architecture](https://blog.cleancoder.com/uncle-bob/2012/08/13/the-clean-architecture.html) falando sobre as várias propostas de arquiteturas:

> Embora todas essas arquiteturas variem um pouco em seus detalhes, elas são muito semelhantes. Todas têm o mesmo objetivo, que é a separação de responsabilidades. Todas alcançam essa separação ao dividir o software em camadas. Cada uma tem pelo menos uma camada para regras de negócio e outra para interfaces.
> Cada uma dessas arquiteturas produz sistemas que são:
> 1. Independentes de frameworks...
> 2. Testáveis...
> 3. **Independentes da UI**...
> 4. Independentes de banco de dados...
> 5. Independentes de qualquer agente externo...

Nesse sentido, o nome da arquitetura não é importante. Estamos em busca de um design que respeita a separação das responsabilidades. 

Fazendo uma pesquisa inicial, o canal no YouTube [Coffstack](https://www.youtube.com/watch?v=HxBEuISRp5I) me levou a 2 excelentes artigos que tentaram chegar, de forma genuína, em uma arquitetura de frontend.

- [Clean Architecture on Frontend by Alex Bespoyasov](https://dev.to/bespoyasov/clean-architecture-on-frontend-4311)
- [Client-Side Architecture Basics [Guide] by Khalil Stemmler](https://khalilstemmler.com/articles/client-side-architecture/introduction)

Esses são bons exemplos de como modelar o software de forma profissional e parecem estar no caminho certo. Seguir essas arquiteturas com certeza produzirá um software de qualidade. 

Ambas abordagens, entretanto, ferem os conceitos da arquitetura limpa. Os Use Cases são criados em hooks, estruturas intrisecamente ligadas ao framework, fazendo com que a camada de Aplication conheça a View. Testar esses hooks, por exemplo, exige um runner que implemente os ciclos de vida do React.

Esse não é bem o modelo que eu estou procurando.

No final de 2024, o time do Flutter definiu sua arquitetura recomendada na doc [Architecting Flutter apps](https://docs.flutter.dev/app-architecture). Essa foi a primeira vez que eu vi algo oficial que parecia com o caminho que tinha pensado, a arquitetura **MVVM**.

<PostImage legend="Guide to app architecture">
  <img src="/mvvm-flutter.png" />
</PostImage>

- **View**: uma camada "burra" que renderiza a UI mas não possui nenhuma regra de negócio, apenas comportamento.
- **Model**: encapsula regras de negócio e uso de recursos externos.
- **View Model**: uma camada orquestradora que conecta assuntos de negócio como entidades e chamadas para apis com as necessidades de estado de uma interface.

Mas eu ainda não estava satisfeito. O Flutter é um framework cross-platform utilizado principalmente para aplicações mobile. Ele foi fortemente inspirado no React em sua época pré hooks então o MVVM parece só uma maneira diferente de implementa-los. Perceba que seu nome já deixa explícito a quebra na separação de responsabilidades: 

> **View Model**, uma camada que conhece a **View** e a **Model**.

Para separar as regras de aplicação da UI, a doc ainda recomenda uma camada extra de Use Cases quando a aplicação começa a crescer.

<PostImage legend="Guide to app architecture">
  <img src="/mvvm-use-case-flutter.png" />
</PostImage>

A View Model pareceu muito com a ideia de um gerenciador global de estados. Com os bindings corretos, é possível conectar uma store Redux, por exemplo, a qualquer framework... e isso sim deixa as regras de aplicação separadas da UI (mas nos deixa presos às biblioteca de estados).

Então, por mais que essa abordagem ferisse esse ideal de arquitetura, ela me interessou por cumprir com muitos dos requisitos de um bom design. O MVVM pode produzir uma camada de negócio _testável_, _independente de framework frontend_, e _independente de qualquer agente externo_.

<PostCallout>
  <span>Existe mesmo uma maneira de separar regras de aplicação da UI?</span>
</PostCallout>

Nesse ponto da história, eu já tinha começado a questionar se realmente existia uma arquitetura de frontend que verdadeiramente separa a camada de Aplication/Use Cases da UI. 

Eu já não tinha tanto tempo, pois eu precisava definir a arquitetura dos projetos da empresa que trabalho. Então criei um design similar ao MVVM, mais ou menos na seguinte forma:

```ts
export interface ITodoViewModel {
  todos: Todo[];
  selectedTodoId: string;
  selectedTodo: Todo;
  createTodo(name: string, description: string): void;
  selectTodo(id: string): void;
  checkSelectedTodo(id: string): void;
}
```

```ts
export class TodoModel {
  constructor(
    public id: string,
    public name: string,
    public description: string,
    public checked: string,
  ) {}
  static create(name: string, description: string) {
    return new TodoModel(uuid(), name, description, false);
  }
  check() {
    this.checked = true;
  }
}
```

```ts
import { create } from 'zustand';
import { compute } from 'zustand-computed-state';
const useApp = create<ITodoViewModel>()((set, get) => ({
  todos: [],
  selectedTodoId: undefined,
  ...compute(get, s => ({
    selectedTodo: s.todos.find(t => t.id === s.selectedTodoId),
  })),
  createTodo(name, description) {
    const todo = TodoModel.create(name, description);
    set({ todos: [...get().todos, todo] });
  },
  selectTodo(id) {
    set({ selectedTodoId: id });
  },
  checkSelectedTodo() {
    const { todos, selectedTodoId } = get();
    const newTodos = todos.map(t => {
      if (t.id === selectedTodoId) t.check();
      return t;
    });
    set({ todos: newTodos });
  },
}));
```

```ts
function getApp() {
  return { get state() { return useApp.getState() as ITodoViewModel } };
}
test('create todo and select it', () => {
  const app = getApp();
  expect(app.state.todos).toHaveLength(0);
  expect(app.state.selectedTodo).toBeUndefined();
  app.state.createTodo('Buy milk', 'Go to the store');
  expect(app.state.todos).toHaveLength(1);
  const todo = app.state.todos[0];
  app.state.selectTodo(todo.id);
  expect(app.state.selectedTodo).toBe(todo);
});
test('check selected todo', () => {
  const app = getApp();
  app.state.createTodo('Buy milk', 'Go to the store');
  const todo = app.state.todos[0];
  expect(todo.checked).toBe(false);
  app.state.selectTodo(todo.id);
  app.state.checkSelectedTodo();
  expect(app.state.selectedTodo.checked).toBe(true);
});
```

A View Model toma papel de Application Service ou Use Case controlando o fluxo de dados e as necessidades de estado da UI como o `selectedTodo`, a Model é uma clássica entidade de domínio e a View pode ser qualquer coisa inclusive um test runner.

Uma grande prioridade que tive foi deixar os testes independentes da UI. Perceba que se removermos o ponto de injeção da store (o `getApp`) da equação, o test runner não tem nenhum conhecimento do framework de gerenciamento de estados ou de UI que está sendo utilizado por debaixo dos panos.

Por mais que a arquitetura tenha ficado robusta ela ainda me incomodava em 2 pontos:

1. Estamos presos na biblioteca de gerenciamento de estados (Zustand).
2. A camada Application (View Model) ainda conhece detalhes da UI (`todos`, `selectedTodoId`, `selectedTodo`)

O primeiro ponto foi fácil de esclarecer com um exemplo. Quando criamos uma aplicação backend em que o Controller implementa diretamente um web framework como `ExpressJS`, estamos quebrando com os conceitos de uma arquitetura limpa? 

Claro que sim! Entretando, isso é relevante? 

Quando essa decisão é deliberada, os desenvolvedores estão assumindo o risco de fazer o lock-in em uma ferramenta por saber que modifica-la será improvável. Desde que a camada Application esteja isolada, a arquitetura estará coerente.

É possível criar uma View Model independente de framework. Só é burocrático e sem ergonomia.

```ts
export class TodoViewModel implements ITodoViewModel {
  todos: Todo[] = [];
  selectedTodoId: string | undefined = undefined;
  constructor(private notifyListeners: () => void) {}
  get selectedTodo() {
    return this.todos.find(t => t.id === this.selectedTodoId);
  }
  createTodo(name: string, description: string) {
    const todo = TodoModel.create(name, description);
    this.todos = [...this.todos, todo];
    this.notifyListeners();
  }
  selectTodo(id: string) {
    this.selectedTodoId = id;
    this.notifyListeners();
  }
  checkSelectedTodo() {
    this.todos = this.todos.map(t => {
      if (t.id === this.selectedTodoId) t.check();
      return t;
    });
    this.notifyListeners();
  }
}
```
```ts
import { create } from 'zustand';
export const useApp = create(() => ({
  todos: viewModel.todos,
  selectedTodoId: viewModel.selectedTodoId,
  selectedTodo: viewModel.selectedTodo,
}));
export const todosViewModel = new TodoViewModel(() => {
  useApp.setState({
    todos: viewModel.todos,
    selectedTodoId: viewModel.selectedTodoId,
    selectedTodo: viewModel.selectedTodo,
  });
});
```

O segundo ponto me incomodou por um bom tempo... a View Model conceitualmente só existe conhecendo a View. Em uma arquitetura _purista_, isso não poderia acontecer.

Pesquisei um pouco mais e encontrei a origem do design. Em [Presentation Model](https://martinfowler.com/eaaDev/PresentationModel.html) (mais tarde aprimorado e solidificado como MVVM), o Martin Fowler explica:

> Presentation Model extrai o estado e o comportamento da View para uma classe de modelo que faz parte da apresentação. [...]

> Presentation Model pode interagir com vários objetos de domínio, mas Presentation Model não é uma fachada amigável à GUI para um objeto de domínio específico. Em vez disso, é mais fácil considerar o Presentation Model como uma abstração da View que não depende de um framework de GUI específico.

Depois de ler esse artigo, percebi que um sotware frontend possui atributos intrínsecos. Não estamos escrevendo um compilador, um sistema de arquivos ou uma API REST. Estamos escrevendo uma aplicação que:

- Possui uma tela com elementos visuais bidimensionais.
- Reage a interações do usuário.
- Mantém um estado de apresentação.
- Utiliza um modelo declarativo para renderização dos componentes.

Como disse no início, a literatura considera o frontend como uma camada, porém essa se tornou muito importante. Mesmo assim ela não deixa de ter suas características. No final das contas, no momento em que decidimos criar uma arquitetura para uma aplicação frontend web, estamos nos vinculando fundamentalmente a um tipo específico de software. **Uma aplicação GUI declarativa 2D**.

Essas não são características opcionais, são a essência do que estamos construindo. A arquitetura certa é uma que abraça esses elementos, em vez de tentar abstraí-los.

O MVVM é exatamente essa arquitetura.
